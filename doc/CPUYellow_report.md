## 二、工作流与交互路线图

### 2.1 整体实施策略

我采用了**自上而下、逐步精化**的设计策略，将任务拆分为以下阶段并分配给AI：

**阶段一：指令集学习与Spec生成**
首先让AI阅读MIPS指令系统规范文档，理解57条指令的编码格式、功能和例外行为。然后指导AI生成完整的设计Spec文档（CPUYellow_spec.md），包含数据通路图、控制信号真值表、模块接口定义和时序要求。这一步的关键在于让AI建立起对整个系统的完整认知。

**阶段二：模块划分与接口设计**
基于Spec文档，要求AI将处理器划分为独立的流水级模块（IF/ID/EX/MEM/WB）以及辅助模块（RegFile/ALU/Control/Hazard/Forwarding/CP0），明确每个模块的输入输出接口和内部逻辑。

**阶段三：逐模块RTL编码**
按照从叶子模块到顶层模块的顺序，指导AI逐个生成RTL代码。顺序为：ALU → RegFile → Control Unit → Hazard/Forwarding → 各流水级 → 顶层集成。每个模块生成后检查接口一致性和时序正确性。

**阶段四：Testbench与验证**
要求AI生成SystemVerilog测试平台，包含指令存储器/数据存储器的行为模型，以及逐条指令的测试用例。每个测试用例包含明确的期望值和自动化的Pass/Fail判断。

**阶段五：报告生成**
最后指导AI生成设计Spec、验证报告和本总结报告。

### 2.2 Agent能力适配与优化

在本次设计过程中，AI表现出以下特性：

**优势：**
- **代码生成能力强**: AI能够快速生成大量格式正确、风格统一的Verilog代码
- **指令集理解准确**: 对于MIPS指令的编码格式和功能描述理解准确
- **文档生成高效**: 能够快速生成结构完整的设计文档

**弱点与改进策略：**
- **时序逻辑容易出错**: AI生成的状态机和流水线寄存器更新逻辑需要仔细检查。改进方法是在提示词中明确给出时序波形图或时序表，让AI理解"哪个信号在哪个时钟沿更新"
- **模块间接口不一致**: 不同模块的端口名称和位宽容易不匹配。改进方法是先生成完整的接口定义表，再让AI严格对照接口表进行编码
- **分支预测逻辑复杂**: Always Taken策略的实现细节需要多次迭代。改进方法是在提示词中提供具体的时序示例（如Task文档中给出的上电第一条指令即为分支指令的例子）
- **控制信号生成不完整**: 初始版本的控制单元遗漏了部分指令的控制信号。改进方法是提供完整的控制信号真值表作为输入

### 3.4 关键问题描述

**问题一：如何实现数据冒险检测？**

数据冒险检测通过两个模块协同实现：

1. **Forwarding Unit（前递单元）**：在EX阶段检测RAW（Read-After-Write）冒险。检测逻辑为：
   - 若EX/MEM阶段的reg_write=1且write_reg匹配当前ID/EX的rs或rt，则从EX/MEM前递（优先级高）
   - 若MEM/WB阶段的reg_write=1且write_reg匹配当前ID/EX的rs或rt，则从MEM/WB前递（优先级低）
   - 前递数据通过一个2选1多路选择器替换ALU的输入操作数

2. **Hazard Detection Unit（冒险检测单元）**：专门处理Load-Use冒险。当ID/EX阶段的指令是Load指令，且其目标寄存器匹配IF/ID阶段下一条指令的rs或rt时，插入一个Stall周期：
   - PC保持不变（不取新指令）
   - IF/ID寄存器保持不变（重复执行ID阶段）
   - ID/EX寄存器被清零（插入NOP气泡）

优化处理覆盖了以下情况：
- EX→EX前递（1-cycle RAW）
- MEM→EX前递（2-cycle RAW）
- Load-Use Stall（不可避免的1-cycle停顿）

**问题二：分支指令是否使用了优化结构？如何进行分支预测？**

是的，使用了Always Taken分支预测策略。设计实现如下：

- **预测策略**：在ID阶段解码出分支指令后，默认预测所有分支指令都跳转（is_taken=1）
- **延迟槽支持**：根据MIPS规范，分支指令后的延迟槽指令必须执行。编译时在分支指令后插入NOP作为延迟槽
- **设计位置**：分支预测逻辑分布在：
  - ID阶段（id_stage.v）：计算分支目标地址，生成is_taken和branch_target信号
  - IF阶段（if_stage.v）：根据is_taken选择next_pc为branch_target或pc+4
  - EX阶段（ex_stage.v）：分支解析，检测预测错误（branch_mispredict信号）
- **预测错误恢复**：当BEQ/BNE在EX阶段判断实际不跳转时，branch_mispredict=1，流水线需要冲刷IF/ID阶段的错误指令

**问题三：对哪些例外进行了处理？如何支持中断？**

处理器实现了以下例外处理：

1. **SystemCall (ExcCode=8)**：由SYSCALL指令触发
2. **Breakpoint (ExcCode=9)**：由BREAK指令触发
3. **Overflow (ExcCode=12)**：由ADD/SUB/ADDI指令溢出触发
4. **Reserved Instruction (ExcCode=10)**：由未定义指令触发
5. **Address Error Load (ExcCode=4)**：由未对齐的LW/LH访问触发
6. **Address Error Store (ExcCode=5)**：由未对齐的SW/SH访问触发

例外处理的流程：
1. 例外发生时，在MEM阶段检测并传递例外信息
2. CP0模块保存EPC（例外指令PC）并设置Cause寄存器的ExcCode
3. Status.EXL置1，表示进入例外处理模式
4. PC被设置为例外入口地址0xBFC0_0380
5. ERET指令执行时，PC恢复为EPC，Status.EXL清零

中断支持的实现：
- CP0的Cause寄存器预留了IP（Interrupt Pending）位
- Status寄存器实现了IE（Interrupt Enable）位
- 中断信号接口已预留，完整的中断控制器可在后续迭代中添加

**问题四：设计中功能覆盖是如何验证的？**

功能覆盖验证采用了多层次的策略：

1. **指令级覆盖**：为每一条指令编写了至少一个测试用例，确保所有57条指令都被执行并验证
2. **冒险场景覆盖**：专门设计了数据冒险（前递1-cycle/2-cycle、Load-Use Stall）和控制冒险（分支预测正确/错误）的测试场景
3. **边界条件覆盖**：包括溢出检测、除零处理、地址对齐检查、寄存器0硬连线等
4. **例外路径覆盖**：设计了SYSCALL/BREAK/Overflow等例外的测试用例
5. **自动化检查**：Testbench中每个测试用例都包含自动化的Pass/Fail判断，通过check_register()和check_memory()任务自动比对期望值和实际值
6. **波形观察**：生成VCD波形文件，可观察PC、指令、ALU结果、写使能、前递控制等关键信号的时序

可使用Verilator的--coverage选项收集代码覆盖率。

## 四、知识盲区识别与整理

### 4.1 体系结构知识盲区

在本次设计中，通过向AI提问，我补充了以下知识盲区：

1. **MIPS延迟槽机制**：之前只知道分支指令后有延迟槽，但不清楚具体实现方式。通过AI的详细解释（"分支指令后的指令无论条件如何都会被执行"）和Task文档中的时序示例，理解了Always Taken预测结合延迟槽的PC更新逻辑。

2. **CP0协处理器的例外处理流程**：最初对EPC的保存时机（延迟槽指令例外时EPC保存的是分支指令的PC）不清楚。通过AI的CP0设计代码，理解了Status.EXL位的作用和ERET的返回机制。

3. **数据前递的优先级**：当EX/MEM和MEM/WB同时匹配同一个源寄存器时，应该选择哪个？AI在设计forwarding_unit时正确处理了这个问题：优先使用EX/MEM的数据（更新的数据）。

4. **Load-Use冒险的不可避免性**：由于数据存储器有1拍读取延迟，即使使用了前递也无法消除Load-Use冒险，必须插入1个Stall周期。AI的解释帮助我理解了为什么不能仅靠前递解决所有数据冒险。

### 4.2 工具与辅助技术知识盲区

1. **Verilator覆盖率收集**：学习了`verilator --coverage`选项的使用方法以及覆盖率报告的分析

2. **SystemVerilog与Verilog的混合使用**：Testbench使用SystemVerilog编写（task/function/initial块等），RTL使用Verilog编写，了解了两种语言的适用场景

3. **Verilog中的$signed()系统函数**：在有符号运算中需要显式使用$signed()进行类型转换，否则Verilog默认进行无符号运算

4. **段间寄存器的时序设计**：理解了为什么IF/ID寄存器需要在stall时保持不变（if_id_write=0），而ID/EX寄存器在stall时需要插入NOP（id_ex_flush=1）

### 4.3 AI生成内容的质疑

在审查AI生成的代码时，我提出了以下几个质疑：

1. **控制信号完整性**：AI初始生成的控制单元遗漏了SLTIU和ANDI等指令的零扩展处理。经过检查后补充了use_zero_ext判断逻辑。

2. **分支预测错误处理的范围**：AI最初对BGEZ/BLTZ的预测错误处理不完整。因为BGEZ/BLTZ的条件判断在ID阶段就能完成（仅需判断GPR[rs]与0的关系），所以预测正确的概率更高。但对于BEQ/BNE，条件判断需要等到EX阶段（两个寄存器值比较），预测错误的概率更大。

3. **HI/LO寄存器的多周期一致性**：在乘法/除法指令后立即使用MFHI/MFLO时，是否会发生数据冒险？分析后发现，HI/LO寄存器不在通用寄存器堆中，不需要通过Forwarding Unit前递，但需要确保HI/LO在多周期乘除法完成前不被读取。当前简化设计中假设单周期完成，在实际硬件中需要额外的互锁机制。

## 五、总结

### 1. 使用基于Agent进行体系结构/数字电路设计的经验和收获

本次实验让我深刻体会到了AI Agent在数字电路设计中的辅助价值。最大的收获是学会了如何编写有效的提示词：一个好的提示词需要包含明确的输入（如指令编码表、接口定义表）、期望的输出格式（如模块模板）、边界条件说明（如时序行为表）和约束条件（如命名规范）。模糊的提示词会导致AI生成不完整或错误的代码，而结构化的提示词能大幅提高生成质量。

另一个重要收获是学会了"验证驱动设计"的思维模式。在AI生成代码后，不能盲目信任，必须通过仿真波形、覆盖率报告和人工代码审查来验证。本实验中，多个模块的接口不匹配和时序错误都是通过这种验证流程发现的。

### 2. Agent辅助下的效率提升

Agent在以下环节显著提升了效率：
- **文档生成**：Spec文档和验证报告的结构化生成节省了约70%的文档编写时间
- **代码框架搭建**：AI生成的模块骨架代码格式统一、注释完整，减少了大量重复性代码的编写
- **指令编码表生成**：57条指令的编码信息手工整理容易出错，AI生成快速准确

效率提升有限甚至需要更多时间的环节：
- **时序调试**：流水线的时序问题仍然需要人工分析和波形观察
- **接口一致性检查**：模块间的端口匹配需要人工逐一核对
- **边界条件处理**：AI对特殊情况的处理常不完整，需要人工补充

### 3. 人的替代性/不可替代性讨论

通过本次实验，我认为在数字电路设计领域：

**AI可以替代的部分**：重复性的代码编写、格式化的文档生成、标准化的接口定义、已知指令集的编码映射——这些工作AI做得又快又好。

**AI不可替代的部分**：
- **架构决策**：选择Harvard还是Von Neumann架构、采用几级流水线、选择哪种分支预测策略——这些需要权衡功耗、面积、性能的决策仍需人工判断
- **时序调试**：理解波形中的毛刺来源、分析建立/保持时间违例——AI目前无法替代人工的调试经验
- **创新性设计**：提出全新的微架构优化方案——AI擅长组合已有知识，但真正的创新仍依赖人类
- **验证完备性判断**：决定"测试到什么程度足够"——这需要工程经验和对应用场景的理解

总的来说，AI Agent是一个强大的设计助手，能显著提高效率，但无法替代工程师的核心判断力和创造力。
